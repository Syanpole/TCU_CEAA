## 🛡️ Security Best Practices

### ✅ DO:
- ✅ Keep `backend/.env` secure (already in `.gitignore`)
- ✅ Use different secrets for staging/production
- ✅ Rotate secrets regularly (every 3-6 months)
- ✅ Use strong passwords (minimum 16 characters)
- ✅ Enable 2FA on AWS and GitHub accounts
- ✅ Review AWS IAM permissions regularly
- ✅ Monitor AWS costs and usage

### ❌ DON'T:
- ❌ Never commit `.env` to Git
- ❌ Never share secrets in Slack/Discord/Email
- ❌ Never use production secrets in staging
- ❌ Never hardcode secrets in code
- ❌ Never expose secrets in logs

## 🔄 Updating Secrets

### When AWS credentials change:

```powershell
# 1. Update backend/.env with new AWS keys
# 2. Regenerate Kubernetes secrets
cd scripts
.\generate-k8s-secrets.ps1 -Environment staging

# 3. Update GitHub secrets
.\sync-github-secrets.ps1

# 4. Apply to Kubernetes cluster
kubectl apply -f ..\k8s\staging\01-secrets.yaml

# 5. Restart pods to pick up new secrets
kubectl rollout restart deployment/backend -n tcu-ceaa-staging
kubectl rollout restart deployment/celery-worker -n tcu-ceaa-staging
```

### When Django SECRET_KEY needs rotation:

```powershell
# 1. Generate new key
python -c "from django.core.management.utils import get_random_secret_key; print(get_random_secret_key())"

# 2. Update backend/.env
# 3. Follow steps above to update K8s and GitHub
```

## 🚨 Troubleshooting

### Issue: "Secret not found" in Kubernetes

**Solution:**
```bash
# Check if secret exists
kubectl get secrets -n tcu-ceaa-staging

# If missing, apply it
kubectl apply -f k8s/staging/01-secrets.yaml

# Verify it was created
kubectl describe secret tcu-ceaa-secrets -n tcu-ceaa-staging
```

### Issue: "Authentication failed" for AWS

**Solution:**
```bash
# Test AWS credentials locally
aws s3 ls s3://tcu-ceaa-staging-media --region us-east-1

# If fails, check:
# 1. AWS_ACCESS_KEY_ID is correct
# 2. AWS_SECRET_ACCESS_KEY is correct
# 3. IAM user has AmazonS3FullAccess policy
# 4. IAM user has AmazonTextractFullAccess policy
```

### Issue: "Email sending failed"

**Solution:**
```bash
# Test email credentials
cd backend
python manage.py shell

from django.core.mail import send_mail
send_mail(
    'Test Subject',
    'Test message',
    'tcu.ceaa.scholarships@gmail.com',
    ['your-email@example.com'],
)

# If fails, check:
# 1. EMAIL_HOST_PASSWORD is Gmail app password (not regular password)
# 2. 2-Step Verification is enabled on Gmail account
# 3. App password was generated correctly
```

### Issue: "Database connection refused"

**Solution:**
```bash
# Check if postgres pod is running
kubectl get pods -n tcu-ceaa-staging

# Check postgres logs
kubectl logs statefulset/postgres -n tcu-ceaa-staging

# Verify POSTGRES_PASSWORD matches in:
# 1. backend/.env
# 2. k8s/staging/01-secrets.yaml
# 3. Backend deployment environment
```

## ❓ FAQ

**Q: Do I need to commit secrets to Git?**  
A: **NO!** Never commit secrets. They're already in `.gitignore`.

**Q: Can I use the same secrets for staging and production?**  
A: **Not recommended.** Use different databases, S3 buckets, and credentials for each environment.

**Q: How often should I rotate secrets?**  
A: Every 3-6 months for production. Immediately if compromised.

**Q: What if I accidentally commit secrets to Git?**  
A: 
1. Immediately rotate ALL secrets
2. Remove from Git history: `git filter-branch` or BFG Repo-Cleaner
3. Update all deployments with new secrets

**Q: Do GitHub secrets sync automatically to Kubernetes?**  
A: No. GitHub secrets are for CI/CD only. Kubernetes secrets must be applied separately.
